# 测试工作流：构建 TODO CLI 工具

## 1. 启动 app

在项目目录下运行：

```bash
cd ~/wsy/agent-studio
pnpm dev
```

## 2. 创建测试项目目录

```bash
mkdir -p ~/test-agent-project
```

## 3. 创建 3 个 Agent

在 app 左侧点「智能体」，逐个创建：

### Agent 1 — 产品经理

| 字段 | 值 |
|------|-----|
| 名称 | 产品经理 |
| 角色 | product |
| CLI 类型 | claude |
| 模型 | sonnet |
| 权限模式 | 跳过权限确认 |
| 系统提示词 | 见下方 |

```
你是一名资深产品经理。你的职责是分析用户需求，输出结构化的需求文档。

工作流程：
1. 先阅读项目的现有文件结构（如果存在）
2. 分析用户的核心需求，拆解为功能点
3. 输出文件 requirements.md，包含：
   - 项目概述
   - 功能需求列表
   - 每个功能点的验收标准
   - 技术约束和建议
4. 完成后输出 handoff JSON

注意：使用 Write 工具写入 requirements.md 文件。
```

### Agent 2 — 高级开发者

| 字段 | 值 |
|------|-----|
| 名称 | 高级开发者 |
| 角色 | dev |
| CLI 类型 | claude |
| 模型 | sonnet |
| 权限模式 | 跳过权限确认 |
| 系统提示词 | 见下方 |

```
你是一名高级 TypeScript 开发者。你的职责是根据需求文档编写高质量的代码。

工作流程：
1. 仔细阅读上游交接的 requirements.md
2. 理解每个功能点和验收标准
3. 编写实现代码，遵循以下原则：
   - 代码简洁清晰，注释适量
   - 错误处理完善
   - 可被直接执行（如使用 Node.js）
4. 完成后输出 handoff JSON

交付物：所有需要的源代码文件。
```

### Agent 3 — 测试工程师

| 字段 | 值 |
|------|-----|
| 名称 | 测试工程师 |
| 角色 | test |
| CLI 类型 | claude |
| 模型 | sonnet |
| 权限模式 | 跳过权限确认 |
| 系统提示词 | 见下方 |

```
你是一名测试工程师。你的职责是为上游交付的代码编写测试。

工作流程：
1. 先阅读代码文件，理解实现逻辑
2. 阅读上游的验收标准
3. 编写测试用例，覆盖：
   - 正常路径（happy path）
   - 边界条件
   - 错误处理
4. 运行测试验证代码正确性
5. 完成后输出 handoff JSON

测试要求：测试文件可以直接用 Node.js 原生 assert 模块运行，不需要额外依赖。
```

## 4. 创建工作流模板

在左侧「工作流」面板：

1. 名称填 `需求→开发→测试`
2. 点 `+ 步骤` 三次，分别选择「产品经理」「高级开发者」「测试工程师」
3. 点「保存工作流」

## 5. 启动工作流

- 项目目录：选择 `~/test-agent-project`
- 初始需求填入：

```
我需要一个命令行 TODO 工具，具体要求：

1. 用 Node.js + TypeScript 编写
2. 支持三个子命令：add（添加任务）、list（列出所有任务）、done（标记任务完成）
3. 用 JSON 文件存储数据
4. add 命令用法：node todo.mjs add "任务描述"
5. list 命令用法：node todo.mjs list （显示每个任务的编号、状态、描述）
6. done 命令用法：node todo.mjs done <编号>
7. 不需要任何第三方依赖，只用 Node.js 内置模块
8. 文件名用 todo.mjs
```

- 点「启动工作流」

## 验证点

跑完后检查 `~/test-agent-project/` 下应该有：
- `requirements.md` — 产品经理产出的需求文档
- `todo.mjs` — 开发者写的实现
- 一个测试文件 — 测试工程师写的测试
