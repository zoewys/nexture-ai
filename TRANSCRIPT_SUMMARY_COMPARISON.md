# Transcript 摘要方案对比

## 三种方案

### 方案 A：纯规则提取（推荐）

```
events: AgentEvent[]
  → 过滤规则（按 kind 选择性保留）
  → 拼接文本
  → 硬截断 3000 tokens
  → 交给反思引擎
```

保留规则：
- `thinking` → 最后 500 字符（推理结论部分）
- `message` → 完整保留（agent 最终输出）
- `tool-call` → 只留 tool name 列表
- `error` → 完整保留
- `system(↳)` → 完整保留（用户输入）

丢弃：
- `message-delta`（已被 message 包含）
- `tool-result`（大段代码/输出，噪音高）
- `stderr`（调试信息）
- `usage`（token 统计）
- `file-changed`（tool-call 里已隐含）

### 方案 B：LLM 摘要

```
events → 拼接完整文本 → 便宜模型总结 300 字 → 交给反思引擎
```

### 方案 C：全量保留

```
events → 全部序列化 → 直接交给反思引擎（10K-50K tokens）
```

## 横向对比

| 维度 | A. 规则提取 | B. LLM 摘要 | C. 全量保留 |
|------|-----------|------------|-----------|
| 信息保真度 | 中（丢弃 tool-result） | 低（LLM 可能遗漏） | 高 |
| 额外 LLM 调用 | 0 次 | 1 次 | 0 次 |
| 额外成本 | $0 | ~$0.005-0.01 | $0（但反思成本高） |
| 延迟 | ~0ms | +2-5 秒 | ~0ms |
| 给反思引擎的 tokens | ~1000-3000 | ~300-500 | ~5000-50000 |
| 反思质量 | 中高 | 中（双重压缩丢信息） | 最高 |
| 实现复杂度 | 低 | 中 | 最低 |
| 可调试性 | 高（规则透明） | 低（摘要黑箱） | 高 |

## 业界选择

| 产品/论文 | 选择 | 理由 |
|-----------|------|------|
| Claude Code Memory | 规则 | 简单可靠 |
| Reflexion | 全量 | 研究环境不在乎成本 |
| MemGPT | LLM 摘要 | 超长对话必须压缩 |
| Stanford GA | 全量+检索过滤 | 每条 observation 本身很短 |
| AutoGen Teachable | 规则 | 只看 user 说的话 |

## 选择方案 A 的理由

1. Agent Studio 的 transcript 结构已知（AgentEvent 联合类型），规则过滤无歧义
2. 反思需要的信息类型确定——输出、用户输入、错误
3. 零额外成本和延迟
4. 可调试——规则透明，反思质量不好可以精确调整

## 边界情况处理

如果 agent 最终 message 超过 3000 tokens：保留开头 1000 + 结尾 1000，中间用 `[...省略 N 字符...]` 标记。
